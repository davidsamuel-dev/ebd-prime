2) VISAO GERAL DAS FUNCIONALIDADES (IMPLEMENTADAS)
========================================================================
2.1 Autenticacao e conta
- Login e logout com API REST (Bearer token em SecureStore).
- Health check antes do login (backend/api/health.php).
- Recuperacao de senha: lookup + e-mail SMTP + pagina web de redefinicao.
- Gestao de conta: dados, troca de login/senha, exclusao.

2.2 Onboarding da escola
- Fluxo multi-etapas (cadastro-escola): termos, telefone, SMS, dados pessoais,
  credenciais e instituicao.
- Registro de congregacao e primeiro administrador via API REST.

2.3 Navegacao e estrutura do app
- Expo Router: tabs Inicio/Geral + carrossel Inicio/Turmas/Cadastros.
- Stack: login, nova aula, sessao, cadastros, historico, configuracoes, etc.

2.4 Turmas e cadastros
- Turmas: listar, criar, editar (store com turma_id) e remover.
- Cadastros: CRUD alunos/professores, busca, inativos com reativacao/exclusao
  (inclusive em lote).

2.5 Aulas, chamada e relatorio
- Escala de aulas: criar, editar data/numero da licao, remover (long press).
- Sessao de aula: presenca, biblia/revista, visitantes, oferta, observacoes.
- Envio de relatorio por turma; frequencia isolada por data/aula.

2.6 Relatorios e dashboards
- Resumo consolidado da aula (aula-resumo.php + telas aula-resumo).
- Painel Geral com endpoint real (geral-resumo.php): rankings, indicadores,
  vencedores por departamento, filtros por periodo.
- Resumo por turma por intervalo (resumo-intervalo.php).
- Historico do aluno, aniversariantes do mes, modulo de ausentes.

2.7 Administracao e site
- Acesso administrativo: listagem e convite de administradores.
- Dados da escola e configuracao SMTP por congregacao.
- Site institucional (landing/) em ebd.adparaiso.com.br com central de ajuda.
- Deploy automatizado para Hostinger via FTP (npm run deploy:hostinger).

2.8 Arquitetura actual (producao)
- REST-only: PHP 8 + PDO + MySQL na Hostinger.
- Firebase/Firestore REMOVIDO do codigo mobile actual; nao e backend activo.
- Multi-igreja na mesma base MySQL com isolamento por congregacao_id.

3) REQUISITOS FUNCIONAIS (RF001…RF050) — ESTADO ACTUAL
========================================================================
Numeracao alinhada a docs/requisitos_ebd_prime.txt

--- 3.1 Autenticacao e sessao ---
RF001  Autenticacao (login, health check, atalhos)
       Status: IMPLEMENTADO
       O utilizador inicia sessao com credenciais validadas pelo servidor.
       Antes do login, o app confirma que a API esta disponivel. Atalhos de
       acesso rapido podem ser guardados localmente no dispositivo.

RF002  Recuperacao de senha (UI + fluxo completo)
       Status: IMPLEMENTADO
       Ecra dedicado no app para solicitar recuperacao por e-mail, com
       validacao de identidade e redirecionamento para redefinicao segura
       da senha.

RF003  Backend REST da recuperacao de senha
       Status: IMPLEMENTADO
       API completa: consulta de utilizador, envio de link por SMTP e
       redefinicao de senha com token de uso unico e prazo de validade.

RF004  Recuperacao via Firebase Auth
       Status: OBSOLETO
       Abordagem da versao anterior com Firebase; o fluxo actual e exclusivamente
       REST com e-mail SMTP.

RF005  Logout
       Status: IMPLEMENTADO
       Encerramento de sessao no cliente e revogacao do token no servidor,
       removendo credenciais guardadas no dispositivo.

RF006  Token de sessao (armazenamento seguro + Bearer)
       Status: IMPLEMENTADO
       Token opaco emitido no login, guardado de forma segura no dispositivo
       e enviado em cada pedido autenticado via cabecalho Authorization.

--- 3.2 Onboarding nova escola ---
RF007  Fluxo guiado de cadastro de escola
       Status: IMPLEMENTADO
       Assistente multi-etapas: termos de uso, telefone, dados pessoais,
       credenciais e dados da instituicao, culminando no registo da
       congregacao e do primeiro administrador.

RF008  Validacao SMS no onboarding
       Status: PARCIAL
       Codigo de verificacao por SMS integrado no fluxo; o gateway SMS externo
       real permanece opcional ou dependente de configuracao.

RF009  Registo Firebase no onboarding
       Status: OBSOLETO
       Substituído pelo registo via API REST na base MySQL centralizada.

--- 3.3 Navegacao ---
RF010  Tabs e carrossel principal

       Inicio, Turmas e Cadastros na area principal do app.

RF011  Menu drawer no Portal Geral

RF012  Stack global de rotas

5) REGRAS DE NEGOCIO (RN001…RN007) — ESTADO ACTUAL
========================================================================
RN001  Isolamento por congregacao (multi-tenant)
       Status: IMPLEMENTADO
       Cada escola acede apenas aos seus proprios dados; o servidor filtra
       todas as operacoes pelo identificador da congregacao autenticada.

RN002  Unificacao da logica no cliente REST
       Status: IMPLEMENTADO
       O app mobile comunica exclusivamente com a API REST; nao ha camada
       Firebase activa nem sincronizacao paralela.

RN003  Rankings por frequencia e pontuacao no Painel Geral
       Status: IMPLEMENTADO
       Rankings calculados a partir das presencas e pontuacoes registadas
       nas sessoes de aula, exibidos no painel geral com dados reais.

RN004  Controle hierarquico por papeis (admin, professor, secretario)
       Status: PARCIAL
       Niveis de acesso existem e restringem algumas accoes, mas regras
       finas por campo ou filial ainda nao estao totalmente definidas.

RN005  Fechamento consolidado condicionado a todas as turmas
       Status: PARCIAL
       Regra desejada: consolidar o dia apenas quando todas as turmas
       enviarem relatorio; actualmente o painel geral desbloqueia apos
       o primeiro relatorio (ver RF027 e RF042).

RN006  Turmas vencedoras por departamento no Geral
       Status: IMPLEMENTADO
       O painel geral destaca a turma vencedora de cada departamento
       com base nos indicadores agregados do periodo seleccionado.

RN007  Papeis duplos (aluno + professor) e conflito de horario
       Status: FUTURO
       Deteccao e tratamento de cadastros com papel duplo e conflitos de
       horario entre turmas ainda nao implementados.
